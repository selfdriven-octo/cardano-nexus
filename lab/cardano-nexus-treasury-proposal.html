// Abstract

What this proposal funds

cardano.nexus is a community-maintained map of who is building in Cardano, what governance is happening, and where help is needed — a shared connective layer the ecosystem can trust without trusting a single owner.

This proposal requests ₳350,000 from the Cardano Treasury to deliver the Minimum Viable Version across four milestones over roughly six months: an open ecosystem directory, identity profiles, a DRep and SPO directory, a learning resource library, a contribution-needs board, verified links, basic trust signals, a community-submitted ecosystem map, and a public API.

Everything is delivered open-source (Apache-2.0) with open schemas, so the work is forkable and cannot be captured by any single operator. Funds are released milestone-by-milestone against published acceptance criteria, and each completed milestone is evidenced on-chain.

The success test for the MVP is deliberately simple: can someone arrive at cardano.nexus and understand the Cardano ecosystem better than before? If yes, it is already valuable.

// The need

The problem it addresses

Cardano has the technical foundation — a public chain, native assets, wallets, identity primitives, governance, and a global community. What's missing is the human, social and contextual layer. Information is scattered across X, Discord, GitHub, forums, governance portals and meetups, and that fragmentation produces recurring, costly problems:

People don't know who is building what
Projects struggle to find collaborators
Newcomers don't know where to start
Governance participants lack shared context
Builders repeat work because prior work is hard to find
Trust is built socially but not represented clearly
Good contributions disappear into timelines
Ecosystem memory is weak; coordination relies on personal networks

// Risk management

Risks & mitigations

Scope creep

Fixed MVP scope with milestone gating. Mitigation: out-of-scope ideas are logged for a future round, not added mid-delivery.

Low adoption

A directory is only useful if populated. Mitigation: seed with existing public data and partner with active communities before launch.

Centralisation drift

A connection layer could become a chokepoint. Mitigation: open schemas, forkability, and a path to progressive decentralisation.

Key-person / delivery risk

Dependence on a small team. Mitigation: open-source from day one, documented build, milestone-based payment so unspent funds are returnable.
